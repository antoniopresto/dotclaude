### Insights Arquiteturais Extraídos

#### 1. Separação Criação vs Execução
```
┌─────────────────────────────────────────────┐
│  CRIAÇÃO (LLM)          │  EXECUÇÃO (Det.)  │
├─────────────────────────┼───────────────────┤
│  • Interpreta NL        │  • Determinístico │
│  • Gera test cases      │  • Estável        │
│  • Entende intenção     │  • Reproduzível   │
└─────────────────────────┴───────────────────┘
```
**Insight**: LLMs são bons para ENTENDER, mas execução deve ser determinística.

#### 2. Multi-Attribute Fingerprinting
- Não dependem de um único localizador (ID, CSS)
- Criam "impressão digital" com: cor, tamanho, posição, texto, contexto, estrutura DOM
- **Aplicação para nós**: assertions podem ter múltiplos critérios de match

#### 3. Hierarquia de Fallback
```
Localizador primário falha
    → Tenta identificadores secundários
        → Usa posicionamento relativo
            → Busca por atributos de texto
                → Navega relacionamentos pai/filho
```
**Aplicação para nós**: assertions com fallback graceful

#### 4. Abordagem "Agentica"
Tendência 2025: AI não apenas EXECUTA testes, mas:
- Descobre casos de teste relevantes
- Auto-repara quando UI muda
- Analisa falhas autonomamente
- Conversa para refinar testes

#### 5. Exemplo de Sintaxe (testRigor)
```
click "Login"
enter "user@email.com" into "Email"
click "Submit"
verify page contains "Welcome"
```

#### 6. Exemplo de Sintaxe (Momentic)
```
"Describe the flow you want to verify in natural language
and Momentic's AI handles the rest"
```

## RESEARCH: AI-Powered Testing Tools (2025)

> Pesquisa realizada via subagentes em 2026-01-06

### Ferramentas Principais Identificadas

| Ferramenta | Tipo | Linguagem Natural | Self-Healing | Diferencial |
|------------|------|-------------------|--------------|-------------|
| **Momentic** | E2E | Runtime interpretation | Sim | Intent-based, não DOM |
| **testRigor** | E2E | NLP puro | Sim | Similar Cucumber mas implementado |
| **Meticulous** | Frontend | Zero-write | N/A | Gera testes do uso real |
| **QA.tech** | E2E | Plain English | Sim | Setup 5min vs 8h manual |
| **Applitools** | Visual | NLP Builder | N/A | 4B telas treinadas |
| **mabl** | Multi | Conversacional | Sim | "Agentic Tester" |

### Insights Arquiteturais Extraídos

#### 1. Separação Criação vs Execução
```
┌─────────────────────────────────────────────┐
│  CRIAÇÃO (LLM)          │  EXECUÇÃO (Det.)  │
├─────────────────────────┼───────────────────┤
│  • Interpreta NL        │  • Determinístico │
│  • Gera test cases      │  • Estável        │
│  • Entende intenção     │  • Reproduzível   │
└─────────────────────────┴───────────────────┘
```
**Insight**: LLMs são bons para ENTENDER, mas execução deve ser determinística.

#### 2. Multi-Attribute Fingerprinting
- Não dependem de um único localizador (ID, CSS)
- Criam "impressão digital" com: cor, tamanho, posição, texto, contexto, estrutura DOM
- **Aplicação para nós**: assertions podem ter múltiplos critérios de match

#### 3. Hierarquia de Fallback
```
Localizador primário falha
    → Tenta identificadores secundários
        → Usa posicionamento relativo
            → Busca por atributos de texto
                → Navega relacionamentos pai/filho
```
**Aplicação para nós**: assertions com fallback graceful

#### 4. Abordagem "Agentica"
Tendência 2025: AI não apenas EXECUTA testes, mas:
- Descobre casos de teste relevantes
- Auto-repara quando UI muda
- Analisa falhas autonomamente
- Conversa para refinar testes

#### 5. Exemplo de Sintaxe (testRigor)
```
click "Login"
enter "user@email.com" into "Email"
click "Submit"
verify page contains "Welcome"
```

#### 6. Exemplo de Sintaxe (Momentic)
```
"Describe the flow you want to verify in natural language
and Momentic's AI handles the rest"
```

### Padrões para Nosso Framework

| De | Para | Aplicação |
|----|------|-----------|
| Self-healing locators | Self-healing assertions | Assertions que adaptam ao contexto |
| Intent-based testing | Intent-based specs | Specs descrevem INTENÇÃO, não implementação |
| Visual AI | Content AI | Verificar qualidade do conteúdo, não só existência |
| Zero-write (Meticulous) | Zero-spec? | Gerar specs do uso real do sistema? |

### Perguntas Emergentes

1. **Como fazer "self-healing" em assertions de pesquisa?**
   - Se "Railnova analysis" não existe, procurar "competitor analysis"?

2. **Modelo agentico para specs?**
   - AI sugere specs baseado no que precisa ser feito?

3. **Separação criação/execução no nosso caso:**
   - LLM cria/interpreta specs
   - Validação é determinística (arquivo existe? conteúdo matches?)

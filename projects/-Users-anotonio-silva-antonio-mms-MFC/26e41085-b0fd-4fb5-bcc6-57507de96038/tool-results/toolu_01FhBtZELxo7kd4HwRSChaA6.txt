   121‚Üí## Step 1.5: System-Level Testability Review (Phase 3 Only)
   122‚Üí
   123‚Üí**Skip this step if Epic-Level Mode.** This step only executes in System-Level Mode.
   124‚Üí
   125‚Üí### Actions
   126‚Üí
   127‚Üí1. **Review Architecture for Testability**
   128‚Üí
   129‚Üí   Evaluate architecture against these criteria:
   130‚Üí
   131‚Üí   **Controllability:**
   132‚Üí   - Can we control system state for testing? (API seeding, factories, database reset)
   133‚Üí   - Are external dependencies mockable? (interfaces, dependency injection)
   134‚Üí   - Can we trigger error conditions? (chaos engineering, fault injection)
   135‚Üí
   136‚Üí   **Observability:**
   137‚Üí   - Can we inspect system state? (logging, metrics, traces)
   138‚Üí   - Are test results deterministic? (no race conditions, clear success/failure)
   139‚Üí   - Can we validate NFRs? (performance metrics, security audit logs)
   140‚Üí
   141‚Üí   **Reliability:**
   142‚Üí   - Are tests isolated? (parallel-safe, stateless, cleanup discipline)
   143‚Üí   - Can we reproduce failures? (deterministic waits, HAR capture, seed data)
   144‚Üí   - Are components loosely coupled? (mockable, testable boundaries)
   145‚Üí
   146‚Üí2. **Identify Architecturally Significant Requirements (ASRs)**
   147‚Üí
   148‚Üí   From PRD NFRs and architecture decisions, identify quality requirements that:
   149‚Üí   - Drive architecture decisions (e.g., "Must handle 10K concurrent users" ‚Üí caching architecture)
   150‚Üí   - Pose testability challenges (e.g., "Sub-second response time" ‚Üí performance test infrastructure)
   151‚Üí   - Require special test environments (e.g., "Multi-region deployment" ‚Üí regional test instances)
   152‚Üí
   153‚Üí   Score each ASR using risk matrix (probability √ó impact).
   154‚Üí
   155‚Üí3. **Define Test Levels Strategy**
   156‚Üí
   157‚Üí   Based on architecture (mobile, web, API, microservices, monolith):
   158‚Üí   - Recommend unit/integration/E2E split (e.g., 70/20/10 for API-heavy, 40/30/30 for UI-heavy)
   159‚Üí   - Identify test environment needs (local, staging, ephemeral, production-like)
   160‚Üí   - Define testing approach per technology (Playwright for web, Maestro for mobile, k6 for performance)
   161‚Üí
   162‚Üí4. **Assess NFR Testing Approach**
   163‚Üí
   164‚Üí   For each NFR category:
   165‚Üí   - **Security**: Auth/authz tests, OWASP validation, secret handling (Playwright E2E + security tools)
   166‚Üí   - **Performance**: Load/stress/spike testing with k6, SLO/SLA thresholds
   167‚Üí   - **Reliability**: Error handling, retries, circuit breakers, health checks (Playwright + API tests)
   168‚Üí   - **Maintainability**: Coverage targets, code quality gates, observability validation
   169‚Üí
   170‚Üí5. **Flag Testability Concerns**
   171‚Üí
   172‚Üí   Identify architecture decisions that harm testability:
   173‚Üí   - ‚ùå Tight coupling (no interfaces, hard dependencies)
   174‚Üí   - ‚ùå No dependency injection (can't mock external services)
   175‚Üí   - ‚ùå Hardcoded configurations (can't test different envs)
   176‚Üí   - ‚ùå Missing observability (can't validate NFRs)
   177‚Üí   - ‚ùå Stateful designs (can't parallelize tests)
   178‚Üí
   179‚Üí   **Critical:** If testability concerns are blockers (e.g., "Architecture makes performance testing impossible"), document as CONCERNS or FAIL recommendation for gate check.
   180‚Üí
   181‚Üí6. **Output System-Level Test Design**
   182‚Üí
   183‚Üí   Write to `{output_folder}/test-design-system.md` containing:
   184‚Üí
   185‚Üí   ```markdown
   186‚Üí   # System-Level Test Design
   187‚Üí
   188‚Üí   ## Testability Assessment
   189‚Üí
   190‚Üí   - Controllability: [PASS/CONCERNS/FAIL with details]
   191‚Üí   - Observability: [PASS/CONCERNS/FAIL with details]
   192‚Üí   - Reliability: [PASS/CONCERNS/FAIL with details]
   193‚Üí
   194‚Üí   ## Architecturally Significant Requirements (ASRs)
   195‚Üí
   196‚Üí   [Risk-scored quality requirements]
   197‚Üí
   198‚Üí   ## Test Levels Strategy
   199‚Üí
   200‚Üí   - Unit: [X%] - [Rationale]
   201‚Üí   - Integration: [Y%] - [Rationale]
   202‚Üí   - E2E: [Z%] - [Rationale]
   203‚Üí
   204‚Üí   ## NFR Testing Approach
   205‚Üí
   206‚Üí   - Security: [Approach with tools]
   207‚Üí   - Performance: [Approach with tools]
   208‚Üí   - Reliability: [Approach with tools]
   209‚Üí   - Maintainability: [Approach with tools]
   210‚Üí
   211‚Üí   ## Test Environment Requirements
   212‚Üí
   213‚Üí   [Infrastructure needs based on deployment architecture]
   214‚Üí

   216‚Üí

   342‚Üí## Step 2: Assess and Classify Risks
   343‚Üí
   344‚Üí### Actions
   345‚Üí
   346‚Üí1. **Identify Genuine Risks**
   347‚Üí
   348‚Üí   Filter requirements to isolate actual risks (not just features):
   349‚Üí   - Unresolved technical gaps
   350‚Üí   - Security vulnerabilities
   351‚Üí   - Performance bottlenecks
   352‚Üí   - Data loss or corruption potential
   353‚Üí   - Business impact failures
   354‚Üí   - Operational deployment issues
   355‚Üí
   356‚Üí2. **Classify Risks by Category**
   357‚Üí
   358‚Üí   Use these standard risk categories:
   359‚Üí
   360‚Üí   **TECH** (Technical/Architecture):
   361‚Üí   - Architecture flaws
   362‚Üí   - Integration failures
   363‚Üí   - Scalability issues
   364‚Üí   - Technical debt
   365‚Üí
   366‚Üí   **SEC** (Security):
   367‚Üí   - Missing access controls
   368‚Üí   - Authentication bypass
   369‚Üí   - Data exposure
   370‚Üí   - Injection vulnerabilities
   371‚Üí
   372‚Üí   **PERF** (Performance):
   373‚Üí   - SLA violations
   374‚Üí   - Response time degradation
   375‚Üí   - Resource exhaustion
   376‚Üí   - Scalability limits
   377‚Üí
   378‚Üí   **DATA** (Data Integrity):
   379‚Üí   - Data loss
   380‚Üí   - Data corruption
   381‚Üí   - Inconsistent state
   382‚Üí   - Migration failures
   383‚Üí
   384‚Üí   **BUS** (Business Impact):
   385‚Üí   - User experience degradation
   386‚Üí   - Business logic errors
   387‚Üí   - Revenue impact
   388‚Üí   - Compliance violations
   389‚Üí
   390‚Üí   **OPS** (Operations):
   391‚Üí   - Deployment failures
   392‚Üí   - Configuration errors
   393‚Üí   - Monitoring gaps
   394‚Üí   - Rollback issues
   395‚Üí
   396‚Üí3. **Score Risk Probability**
   397‚Üí
   398‚Üí   Rate likelihood (1-3):
   399‚Üí   - **1 (Unlikely)**: <10% chance, edge case
   400‚Üí   - **2 (Possible)**: 10-50% chance, known scenario
   401‚Üí   - **3 (Likely)**: >50% chance, common occurrence
   402‚Üí
   403‚Üí4. **Score Risk Impact**
   404‚Üí
   405‚Üí   Rate severity (1-3):
   406‚Üí   - **1 (Minor)**: Cosmetic, workaround exists, limited users
   407‚Üí   - **2 (Degraded)**: Feature impaired, workaround difficult, affects many users
   408‚Üí   - **3 (Critical)**: System failure, data loss, no workaround, blocks usage
   409‚Üí
   410‚Üí5. **Calculate Risk Score**
   411‚Üí
   412‚Üí   ```
   413‚Üí   Risk Score = Probability √ó Impact
   414‚Üí
   415‚Üí   Scores:
   416‚Üí   1-2: Low risk (monitor)
   417‚Üí   3-4: Medium risk (plan mitigation)
   418‚Üí   6-9: High risk (immediate mitigation required)
   419‚Üí   ```
   420‚Üí
   421‚Üí6. **Highlight High-Priority Risks**
   422‚Üí
   423‚Üí   Flag all risks with score ‚â•6 for immediate attention.
   424‚Üí
   425‚Üí7. **Request Clarification**
   426‚Üí
   427‚Üí   If evidence is missing or assumptions required:
   428‚Üí   - Document assumptions clearly
   429‚Üí   - Request user clarification
   430‚Üí   - Do NOT speculate on business impact
   431‚Üí
   432‚Üí8. **Plan Mitigations**
   433‚Üí
   434‚Üí   For each high-priority risk:
   435‚Üí   - Define mitigation strategy
   436‚Üí   - Assign owner (dev, QA, ops)
   437‚Üí   - Set timeline
   438‚Üí   - Update residual risk expectation

   442‚Üí## Step 3: Design Test Coverage
   443‚Üí
   444‚Üí### Actions
   445‚Üí
   446‚Üí1. **Break Down Acceptance Criteria**
   447‚Üí
   448‚Üí   Convert each acceptance criterion into atomic test scenarios:
   449‚Üí   - One scenario per testable behavior
   450‚Üí   - Scenarios are independent
   451‚Üí   - Scenarios are repeatable
   452‚Üí   - Scenarios tie back to risk mitigations
   453‚Üí
   454‚Üí2. **Select Appropriate Test Levels**
   455‚Üí
   456‚Üí   **Knowledge Base Reference**: `test-levels-framework.md`
   457‚Üí
   458‚Üí   Map requirements to optimal test levels (avoid duplication):
   459‚Üí
   460‚Üí   **E2E (End-to-End)**:
   461‚Üí   - Critical user journeys
   462‚Üí   - Multi-system integration
   463‚Üí   - Production-like environment
   464‚Üí   - Highest confidence, slowest execution
   465‚Üí
   466‚Üí   **API (Integration)**:
   467‚Üí   - Service contracts
   468‚Üí   - Business logic validation
   469‚Üí   - Fast feedback
   470‚Üí   - Good for complex scenarios
   471‚Üí
   472‚Üí   **Component**:
   473‚Üí   - UI component behavior

   641‚Üí
   642‚Üí**Probability √ó Impact = Risk Score**

   649‚Üí
   650‚Üí**Threshold**: Scores ‚â•6 require immediate mitigation.

   652‚Üí### Test Level Selection Strategy
   653‚Üí
   654‚Üí**Avoid duplication:**
   655‚Üí
   656‚Üí- Don't test same behavior at E2E and API level

   658‚Üí- Use API tests for complex business logic
   659‚Üí- Use unit tests for edge cases
   660‚Üí
   661‚Üí**Tradeoffs:**
   662‚Üí
   663‚Üí- E2E: High confidence, slow execution, brittle
   664‚Üí- API: Good balance, fast, stable

   367→**Key Findings:**
   368→
   369→1. **InstancedMesh Problem:** Three.js sends ALL instances to GPU regardless of viewport

   375→   - Viewport query for 1000 results ~4ms
   376→   - Perfect for viewport culling queries
   377→
   378→3. **Culling Strategy:**
   379→   - Reorder visible instances to front of buffer
   380→   - Set `mesh.count = visibleCount` (only visible rendered)
   381→   - No new allocations during pan/zoom
   382→
   383→4. **Alternative Libraries:**
   384→   - `@three.ez/instanced-mesh` (InstancedMesh2) - has BVH + per-instance culling
   385→   - `three-mesh-bvh` - accelerated raycasting
   386→   - `flatbush` - 5-8x faster than RBush for static data

  1447→### F122: Fix elastic/jump effect when dragging - COMPLETE
  1448→
  1449→**Problem:**
  1450→Canvas felt elastic or jumped when starting a new drag during/after momentum animation.
  1451→
  1452→**Root Cause Analysis:**
  1453→React state updates are batched/async. When user starts a new drag while momentum is running:
  1454→1. `handlePointerDown` captures `cameraPos` from React state
  1455→2. But React state hasn't been updated yet with latest momentum position
  1456→3. New drag starts from stale position → canvas "jumps" back
  1457→
  1458→**Solution Implemented:**
  1459→1. Added `cameraPosRef` to track camera position synchronously via useRef
  1460→2. Created `updateCameraPos()` helper that updates BOTH ref and state
  1461→3. Updated all `setCameraPos()` calls to use `updateCameraPos()`:
  1462→   - `handleCameraInitialized`
  1463→   - `animateZoomTo` (zoom animation loop)
  1464→   - `startMomentum` (momentum animation loop)
  1465→   - `handlePinchZoom`
  1466→   - `handlePointerMove` (pan movement)
  1467→   - `handleDoubleClick` (double-click zoom animation)
  1468→4. Changed `handlePointerDown` to read from `cameraPosRef.current` instead of `cameraPos` state

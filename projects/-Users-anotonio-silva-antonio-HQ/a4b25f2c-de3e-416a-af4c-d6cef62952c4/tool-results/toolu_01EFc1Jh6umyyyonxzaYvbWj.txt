   240→================================================================================
   241→TASK: F011 - Dev Bridge (Browser-to-Native RPC)
   242→================================================================================
   243→
   244→## Problem Statement
   245→
   246→Tauri's `invoke()` function only works inside the native webview context
   247→(when the app runs via `bun tauri dev` and opens in the Tauri window).
   248→
   249→When testing via Playwright MCP, the browser loads `localhost:5173` directly,
   250→which is the Vite dev server WITHOUT Tauri's native layer. This means:
   251→
   252→### What DOES NOT work in browser mode:
   253→- `@tauri-apps/api/core` invoke() - TypeError: Cannot read 'invoke'
   254→- PTY process spawning (pty_spawn, pty_write, pty_kill)
   255→- File system operations (read/write project files)
   256→- Native dialogs (file picker, save dialog)
   257→- System notifications
   258→- Any Rust backend functionality
   259→
   260→### What DOES work in browser mode:
   261→- React components and UI rendering
   262→- State management (@zazcart/tools)
   263→- Mock data and simulated interactions
   264→- PatternFly components
   265→- Pure frontend logic
   266→
   267→## Why This Matters
   268→
   269→1. **Testing Gap**: Cannot verify PTY, file watching, or native features via Playwright
   270→2. **Development Friction**: Must manually test in Tauri window for native features
   271→3. **CI/CD Limitation**: Automated tests cannot cover critical functionality
   272→
   273→## Proposed Solution: Dev Bridge
   274→
   275→Create a development-only RPC bridge that:
   276→
   277→```
   278→┌─────────────────┐     WebSocket/HTTP      ┌─────────────────┐
   279→│  Browser        │ ◄─────────────────────► │  Rust Dev       │
   280→│  (Playwright)   │     localhost:9876      │  Server         │
   281→│                 │     + auth token        │                 │
   282→│  TypeScript     │                         │  Mirrors Tauri  │
   283→│  RPC Client     │                         │  Commands       │
   284→└─────────────────┘                         └─────────────────┘
   285→```
   286→
   287→### Architecture
   288→
   289→1. **Rust Dev Server** (only compiled in debug mode):
   290→   - Binds to localhost:9876 (or configurable port)
   291→   - Accepts WebSocket connections with token auth
   292→   - Exposes same commands as Tauri invoke (pty_spawn, etc.)
   293→   - Streams events back to client (pty-output, pty-exit)
   294→
   295→2. **TypeScript Client** (unified API):
   296→   ```typescript
   297→   // Auto-detects context and routes appropriately
   298→   async function invokeCommand(cmd: string, args: object) {
   299→     if (isTauri()) {
   300→       return await invoke(cmd, args);  // Native path
   301→     } else if (isDevBridge()) {
   302→       return await rpcClient.call(cmd, args);  // Bridge path
   303→     } else {
   304→       throw new Error('No backend available');
   305→     }
   306→   }
   307→   ```
   308→
   309→3. **Security Measures**:
   310→   - localhost-only binding (127.0.0.1, not 0.0.0.0)
   311→   - Token generated at startup, passed via env/file
   312→   - Disabled entirely in release builds (#[cfg(debug_assertions)])
   313→   - Rate limiting to prevent abuse
   314→   - Command allowlist (no arbitrary execution)
   315→
   316→### Future SDK Consideration
   317→
   318→The RPC pattern established here can later be extended for plugins:
   319→- Plugins run in sandboxed context (iframe, worker, or separate process)
   320→- Communicate with host via same RPC protocol
   321→- Host controls which commands plugins can invoke
   322→- No need to redesign later; just add permission layer
   323→
   324→**IMPORTANT**: Do NOT over-engineer for plugins now. Just ensure the bridge
   325→design doesn't preclude adding a permission/sandbox layer later.
   326→
   327→## Implementation Steps
   328→
   329→See F011 steps in features.json. Estimated complexity: Medium-High.

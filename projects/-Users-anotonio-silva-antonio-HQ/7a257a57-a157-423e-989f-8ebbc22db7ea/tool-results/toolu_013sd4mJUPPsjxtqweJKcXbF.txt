    11‚Üí
    12‚Üí### What is a Harness?
    13‚Üí
    14‚ÜíA **Harness** is a framework that enables AI agents to work on long tasks across multiple context windows/sessions.
    15‚Üí
    16‚Üí**Problem:** Each new Claude Code session starts with NO memory of previous sessions.
    17‚Üí
    18‚Üí**Solution:** State files persist context between sessions:
    19‚Üí- `.harness/progress.txt` ‚Üí session history
    20‚Üí- `.harness/features.json` ‚Üí requirements with status (passes: true/false)
    21‚Üí
    22‚Üí### IMPORTANT: 
    23‚Üí - NEVER LEAVE MOCKS IN THIS PROJECT, ALL TASKS MUST BE DELIVERED PRODUCTION READY.
    24‚Üí - NEVER LEAVE TECHNICAL DEBIT IN PROJECT, ALL TASKS MUST BE DELIVERED PRODUCTION READY.
    25‚Üí - ALL TASKS MUST BE DELIVERED PRODUCTION READY.
    26‚Üí
    27‚Üí#### Three Distinct Concepts
    28‚Üí
    29‚Üí| Concept | Description |
    30‚Üí|---------|-------------|
    31‚Üí| **Harness (method)** | The pattern itself - state files for AI continuity |
    32‚Üí| **HQ in development** | THIS project is being BUILT using Harness (`.harness/` here) |
    33‚Üí| **HQ as product** | Once complete, HQ will MANAGE other Harness-based projects |
    34‚Üí
    35‚ÜíThe `.harness/` folder in this repo is for building HQ. The final product will open `.harness/` folders from OTHER projects.
    36‚Üí
    37‚Üí### Work Protocol
    38‚Üí
    39‚Üí```
    40‚Üí0. ALWAYS commit even small progress to save history
    41‚Üí1. READ .harness/progress.txt (understand current state)
    42‚Üí2. READ .harness/features.json (find next feature with passes: false)
    43‚Üí3. IMPLEMENT one feature
    44‚Üí4. TEST visually (bun tauri dev + Playwright MCP)
    45‚Üí5. COMMIT after passing
    46‚Üí6. UPDATE .harness/progress.txt
    47‚Üí7. STOP (leave codebase production-ready)
    48‚Üí```

   103‚Üí## KISS Philosophy
   104‚Üí
   105‚Üí**KISS is NOT "do the minimum". It IS "don't add complexity that blocks progress".**
   106‚Üí
   107‚ÜíThe goal is **speed of delivery with quality**. The product must impress the client to win contracts.
   108‚Üí
   109‚Üí### What KISS means (examples)
   110‚Üí
   111‚Üí| ‚úÖ DO                                                         | ‚ùå DON'T                                                          |
   112‚Üí|--------------------------------------------------------------|------------------------------------------------------------------|
   113‚Üí| Add polish, debug tools, UX refinements that accelerate work | Spend days on infrastructure before having something presentable |
   114‚Üí| Use best practices that prevent future problems              | Add abstractions for hypothetical future needs                   |
   115‚Üí| Preserve knowledge/tips for when they become relevant        | Ignore proven patterns just because "not needed yet"             |
   116‚Üí
   117‚Üí### When to Increase Complexity (examples)
   118‚Üí
   119‚Üí| ‚úÖ Increase complexity when...     | ‚ùå Do NOT increase when... |
   120‚Üí|-----------------------------------|---------------------------|
   121‚Üí| Current approach blocks a feature | "It would be cleaner"     |
   122‚Üí| Performance is unacceptable       | "Best practices say so"   |
   123‚Üí| Code duplication becomes painful  | Premature optimization    |
   124‚Üí| Team needs shared abstractions    | Speculative future needs  |
   125‚Üí
   126‚Üí### Key Rule
   127‚Üí
   128‚Üí**Do NOT write throwaway code without clear time/agility gains.** If code will be rewritten anyway, the simple version must provide faster iteration or validation‚Äîotherwise, just build it right the first time.
   129‚Üí**If you know something will cause problems later (performance, UX, maintainability), address it now based on knowledge and testing‚Äînot fear, not speculation.**

   141‚Üí### State Management: AVOID React Hooks, USE @zazcart/tools (private package)
   142‚Üí
   143‚Üí> **CRITICAL RULE:** Avoid `useState`/`useRef` for state. Use `@zazcart/tools` (private package) state-machine for ALL state.
   144‚Üí
   145‚Üí**Why?**
   146‚Üí- React hooks cause "stale closures" in animation loops
   147‚Üí- State updates are batched/async - values become stale
   148‚Üí- Debugging is hard when state is scattered across multiple hooks
   149‚Üí
   150‚Üí**Exceptions (use hooks only for):**
   151‚Üí- `useCallback`/`useMemo` for render optimization
   152‚Üí- `useEffect` for side effects (event listeners, cleanup)
   153‚Üí- `useRef` ONLY for DOM references (`containerRef`)
   154‚Üí- 100% local disposable UI state (e.g., temporary tooltip position)

   278‚Üí## MCP VISUAL TESTING (MANDATORY)
   279‚Üí
   280‚Üí### Why This is Required
   281‚Üí
   282‚ÜíYou cannot see the app you're building. Without visual testing, you will:
   283‚Üí- Mark blank screens as "working"
   284‚Üí- Miss CSS/layout issues
   285‚Üí- Overlook runtime errors visible only in UI
   286‚Üí
   287‚Üí### Testing Protocol: Playwright MCP
   288‚Üí
   289‚ÜíUse **Playwright MCP** for visual tests. Runs headless and **does not steal focus** from the user's window.
   290‚Üí
   291‚Üí#### Step 1: Navigate to the page
   292‚Üí
   293‚Üí```
   294‚Üímcp__playwright__browser_navigate url="file:///full/path/to/file.html"
   295‚Üí```
   296‚Üí
   297‚ÜíOr for local server:
   298‚Üí```
   299‚Üímcp__playwright__browser_navigate url="http://localhost:5173"
   300‚Üí```
   301‚Üí
   302‚Üí#### Step 2: Check console (JS errors)
   303‚Üí
   304‚Üí```
   305‚Üímcp__playwright__browser_console_messages level="error"
   306‚Üí```
   307‚Üí
   308‚Üí- Red errors ‚Üí **FAIL** (investigate before continuing)
   309‚Üí- Yellow warnings ‚Üí Evaluate (e.g., "Tailwind CDN production warning" is OK)
   310‚Üí
   311‚Üí#### Step 3: Capture screenshot
   312‚Üí
   313‚Üí```
   314‚Üímcp__playwright__browser_take_screenshot filename="screenshots/test-name.png"
   315‚Üí```
   316‚Üí
   317‚ÜíFor scrollable pages:
   318‚Üí```
   319‚Üímcp__playwright__browser_take_screenshot fullPage=true filename="screenshots/test-full.png"
   320‚Üí```
   321‚Üí
   322‚Üí#### Step 4: Analyze visually
   323‚Üí
   324‚ÜíCheck in the screenshot:
   325‚Üí- [ ] Content visible? (not blank/empty)
   326‚Üí- [ ] Layout correct? (elements positioned as expected)
   327‚Üí- [ ] Colors correct? (matching design)
   328‚Üí- [ ] No visual artifacts?
   329‚Üí
   330‚Üí#### Step 5: Interactions (if needed)
   331‚Üí
   332‚Üí```
   333‚Üímcp__playwright__browser_click element="button X" ref="e123"
   334‚Üímcp__playwright__browser_type element="field Y" ref="e456" text="value"
   335‚Üí```
   336‚Üí
   337‚Üí#### Step 6: Close browser at the end
   338‚Üí
   339‚Üí```
   340‚Üímcp__playwright__browser_close
   341‚Üí```
   342‚Üí
   343‚Üí### Approval Criteria
   344‚Üí
   345‚Üí| Criterion              | PASS | FAIL                      |
   346‚Üí|------------------------|------|---------------------------|
   347‚Üí| Console without errors | ‚úÖ    | ‚ùå Red errors              |
   348‚Üí| Content rendered       | ‚úÖ    | ‚ùå Blank/empty screen      |
   349‚Üí| Layout correct         | ‚úÖ    | ‚ùå Misaligned elements     |
   350‚Üí| JS executed            | ‚úÖ    | ‚ùå Dynamic content missing |

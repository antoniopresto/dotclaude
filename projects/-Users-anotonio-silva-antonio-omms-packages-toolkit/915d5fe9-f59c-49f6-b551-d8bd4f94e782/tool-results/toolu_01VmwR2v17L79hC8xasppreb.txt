# Long-Running Agent Harness Methodology

## 1. Initializer Agent
The initializer agent runs once at the start and sets up the foundational environment. It creates three key artifacts: an `init.sh` script, a `claude-progress.txt` file documenting prior work, and an initial git commit showing added files.

## 2. Features File Structure
The initializer creates a comprehensive JSON file listing all required features. Each feature entry includes:
- Category (e.g., "functional")
- Description with specific user actions
- Step-by-step verification steps
- A `passes` boolean field (initially `false`)

The document notes: *"It is unacceptable to remove or edit tests because this could lead to missing or buggy functionality."* Features should number in the hundreds for complex applications.

## 3. Baby Steps Commits
After each session, the coding agent commits progress with descriptive messages and writes progress summaries. This enables reverting failed changes and maintaining working codebases between context windows.

## 4. Test-Driven Development Approach
Rather than unit testing alone, agents must verify features end-to-end as humans would. The methodology recommends using browser automation tools (like Puppeteer) to test web applications through actual user interactions—clicking buttons, submitting forms, and observing results.

## 5. Multi-Context Window Workflows
Each new session follows a consistent startup pattern: check current directory, read progress notes and git logs, review the features list, start the development server, run basic smoke tests, then tackle one uncompleted feature.

## 6. File Structures & Templates
Key files include:
- `init.sh` - Development server startup script
- `claude-progress.txt` - Session history log
- `feature_list.json` - Structured feature tracking (JSON preferred over Markdown)
- Git history with clear commit messages

## 7. Best Practices for Long-Running Agents
- **Incremental progress**: Work on one feature per session to prevent context exhaustion
- **Environment validation**: Always verify the app runs before implementing new features
- **Clean state**: Leave code ready for merging to production branches
- **Explicit testing prompts**: Model requires clear directives to perform thorough verification
- **Artifact-based handoff**: Progress files and git history bridge memory gaps between sessions

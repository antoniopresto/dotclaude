# Claude Code Best Practices Summary

## 1) Project Structure

**Key Recommendation:**
Place `CLAUDE.md` files at multiple levels:
- Root directory (checked into git for team sharing)
- Parent directories (useful for monorepos)
- Child directories (pulled on-demand)
- Home folder `~/.claude/CLAUDE.md` (applies globally)

Store custom slash commands in `.claude/commands/` folder and MCP configurations in `.mcp.json` files.

## 2) CLAUDE.md Files

**Purpose:** "a special file that Claude automatically pulls into context when starting a conversation"

**Recommended Content:**
- Common bash commands with descriptions
- Core files and utility function references
- Code style guidelines
- Testing and workflow instructions
- Repository conventions (branching, merging)
- Developer environment setup requirements
- Project-specific quirks or warnings

**Optimization Strategy:** Treat CLAUDE.md files like prompts—refine iteratively based on Claude's adherence. Press `#` to have Claude automatically incorporate new instructions. Run files through the prompt improver tool periodically.

## 3) Test-Driven Development

**Workflow Steps:**
1. Request tests based on input/output pairs; explicitly note you're doing TDD
2. Verify tests fail without implementation
3. Commit passing tests
4. Request implementation code that passes tests only
5. Run tests iteratively until all pass
6. Optionally verify with independent subagents against overfitting
7. Commit final code

**Advantage:** Provides "a clear target to iterate against," enabling Claude to make changes, evaluate results, and incrementally improve.

## 4) Commit Strategies

**Recommended Approach:**
- Have Claude write commit messages automatically; it reads diffs and recent history
- Use Claude for complex git operations: reverting files, resolving rebases, comparing patches
- Commit at natural boundaries: after tests pass, after implementation phases, after planning approval
- Include CLAUDE.md refinements in commits so team members benefit

## 5) Multi-Agent Workflows

**Three Primary Patterns:**

**Sequential Review:**
- One Claude writes code
- Second Claude reviews or tests it
- Third Claude refines based on feedback

**Parallel Task Distribution:**
- Create 3-4 git worktrees with `git worktree add ../project-feature branch-name`
- Launch separate Claude instances per worktree for independent tasks
- Avoids conflicts and allows simultaneous progress

**Headless Mode Integration:**
- Use `claude -p` for non-interactive contexts (CI, pre-commit hooks)
- Implement "fanning out" for large migrations (generate task list, process each via Claude)
- Implement "pipelining" to integrate Claude into data processing workflows

## 6) File Structure Templates

**Recommended Directory Organization:**

```
project-root/
├── CLAUDE.md                 # Shared team guidelines
├── CLAUDE.local.md          # Personal, gitignored version
├── .mcp.json                # MCP server configuration
├── .claude/
│   ├── settings.json        # Permission allowlist (checked in)
│   └── commands/
│       ├── fix-github-issue.md
│       └── [other-templates].md
└── [source code & tests]
```

**Custom Command Template Example:**
```
.claude/commands/[workflow].md
- Use special keyword $ARGUMENTS to pass parameters
- Include step-by-step instructions
- Reference available CLI tools and scripts
```

## Additional Optimization Tips

- **Permissions Management:** Use `/permissions` to allowlist safe tools, reducing approval prompts
- **Context Efficiency:** Use `/clear` between tasks to prevent irrelevant content from cluttering context
- **Visual Collaboration:** Include screenshots, design mocks, and images for UI/UX tasks
- **Specificity Matters:** Detailed instructions significantly improve first-attempt success rates

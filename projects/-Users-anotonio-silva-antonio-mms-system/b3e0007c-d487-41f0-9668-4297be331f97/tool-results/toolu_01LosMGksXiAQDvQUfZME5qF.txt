    58→## Playwright MCP Verification Workflow
    59→
    60→> **MANDATORY** for all UI work (new features AND verification of completed features)
    61→
    62→### Step-by-step E2E verification using Playwright MCP:
    63→
    64→**1. Start dev server** (in separate terminal):
    65→```bash
    66→PORT=0 bun run dev
    67→```
    68→> **CRITICAL**: Capture the actual port from server output (e.g., "Local: http://localhost:43521")
    69→
    70→**2. Navigate with Playwright MCP**:
    71→- Use `mcp__playwright__browser_navigate` to `http://localhost:<captured-port>`
    72→- **Never hardcode ports** - always use the dynamically assigned port
    73→
    74→**3. Capture page state**:
    75→- Use `mcp__playwright__browser_snapshot` to get accessibility tree
    76→- Review snapshot for expected elements
    77→
    78→**4. Verify interactions**:
    79→- Use `mcp__playwright__browser_click` for clicks
    80→- Use `mcp__playwright__browser_type` for text input
    81→- Use `mcp__playwright__browser_press_key` for keyboard shortcuts
    82→
    83→**5. Validate expected state**:
    84→- Take another snapshot after interactions
    85→- Verify expected changes occurred
    86→
    87→**6. Cleanup**:
    88→- Use `mcp__playwright__browser_close`
    89→
    90→### If Playwright MCP verification fails:
    91→- Task is **NOT** complete
    92→- Document issue in @IMPLEMENTATION_PLAN.md
    93→- Fix before proceeding
    94→
    95→### Key Playwright MCP Tools:
    96→| Tool | Purpose |
    97→|------|---------|
    98→| `mcp__playwright__browser_navigate` | Navigate to URL |
    99→| `mcp__playwright__browser_snapshot` | Capture accessibility tree |
   100→| `mcp__playwright__browser_click` | Click element by ref |
   101→| `mcp__playwright__browser_type` | Type text into element |
   102→| `mcp__playwright__browser_press_key` | Press keyboard key |
   103→| `mcp__playwright__browser_take_screenshot` | Capture visual screenshot |
   104→| `mcp__playwright__browser_close` | Close browser |
   105→
   106→### Verification Tasks (Phase 7.8):
   107→- Completed UI features must be verified with Playwright MCP
   108→- New UI features must pass Playwright MCP before being marked complete
   109→- See IMPLEMENTATION_PLAN.md Phase 7.8 for full task list

   218→## Codebase Patterns
   219→
   220→### Domain Services
   221→- Location: `src/services/`
   222→- Pattern: Class with injected dependencies
   223→- All services emit OpenTelemetry traces
   224→- Use Drizzle ORM for database access
   225→
   226→### API Routes
   227→- Location: `src/api/routes/`
   228→- Framework: Hono
   229→- All routes have OpenAPI spec
   230→- Use @belt/tools createSchema for validation
   231→
   232→### Workers
   233→- Location: `src/workers/`
   234→- Framework: BullMQ (Redis-backed)
   235→- Central exports: `src/workers/index.ts`
   236→- Use `registerWorker()` to create new workers
   237→- 8 queues: ingestion, tactical_scheduler, strategic_planner, optimizer, manhour_leveling, drift_monitor, notification, report
   238→
   239→**Running Workers:**
   240→```bash
   241→# Start all workers
   242→bun run src/workers/runner.ts
   243→
   244→# Start specific workers only
   245→WORKERS=optimizer,drift_monitor bun run src/workers/runner.ts
   246→
   247→# Workers require Redis to be running
   248→docker-compose up -d redis
   249→```
   250→
   251→**Implemented Processors (All 9 production-ready):**
   252→- `optimizer` - Run assignment optimization (C++ OR-Tools or TS fallback)
   253→- `drift_monitor` - Baseline drift analysis
   254→- `ingestion` - CSV mileage file processing
   255→- `maximo_sync` - Bidirectional sync with Maximo (push/pull/full)
   256→- `tactical_scheduler` - PM scheduling for 12-week horizon
   257→- `strategic_planner` - Long-term PM planning (genetic algorithm)
   258→- `manhour_leveling` - Workforce optimization across depots
   259→- `notification` - Email notification delivery
   260→- `report` - Excel/PDF report generation
   261→
   262→### Optimizer IPC
   263→- Protocol: NDJSON over stdin/stdout
   264→- Input: `{ type: "solve", data: {...} }`
   265→- Output: `{ type: "progress" | "solution" | "error", ... }`

   267→### Validation & State Management (@belt/tools)
   268→
   269→> **CRITICAL**: See README.md for complete @belt/tools API reference
   270→
   271→**Schema Validation** (replaces Zod):
   272→```typescript
   273→import { createSchema, createType, InferType } from '@belt/tools';
   274→
   275→const TrainModel = createSchema({
   276→  id: 'string',
   277→  trainNumber: 'string',
   278→  mileageKm: 'number',
   279→  fitnessStatus: { enum: ['FIT', 'RESTRICTED', 'GROUNDED'] },
   280→});
   281→type Train = InferType<typeof TrainModel>;
   282→const train = TrainModel.parse(data);
   283→```
   284→
   285→**State Management** (replaces React hooks + zustand):
   286→```typescript
   287→import { createSlice, createMachine } from '@belt/tools';
   288→
   289→const fleetSlice = createSlice({
   290→  id: 'fleet',
   291→  initial: { trains: [], selected: null },
   292→  cache: { policy: 'session' },
   293→  reducers: {
   294→    setTrains: (state, trains) => { state.trains = trains; },
   295→  },
   296→});
   297→
   298→const machine = createMachine({
   299→  id: 'mms',
   300→  slices: { fleet: fleetSlice },
   301→  trackMetadata: false, // Critical for 60fps
   302→});
   303→
   304→// In React component (NO useState/useEffect!)
   305→const { state, actions } = machine.useMachine('fleet');
   306→```

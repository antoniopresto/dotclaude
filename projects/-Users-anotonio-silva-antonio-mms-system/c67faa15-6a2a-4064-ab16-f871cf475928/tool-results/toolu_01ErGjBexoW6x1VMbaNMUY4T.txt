   434вҶ’### 2.4.4 Error Handling Protocol
   435вҶ’
   436вҶ’**CRITICAL:** The C++ optimizer communicates via stdin/stdout streams. Stream failures (segfaults, hangs) must be detectable by the TypeScript worker.
   437вҶ’
   438вҶ’```typescript
   439вҶ’// Worker-side error handling (TypeScript)
   440вҶ’class OptimizerWorker {
   441вҶ’  private process: ChildProcess;
   442вҶ’  private heartbeatTimeout: NodeJS.Timeout | null = null;
   443вҶ’  private lastHeartbeat: number = 0;
   444вҶ’  
   445вҶ’  private readonly HEARTBEAT_INTERVAL_MS = 5000;
   446вҶ’  private readonly HEARTBEAT_TIMEOUT_MS = 15000;  // 3 missed heartbeats
   447вҶ’  
   448вҶ’  async solve(request: OptimizerInput): Promise<OptimizerOutput> {
   449вҶ’    return new Promise((resolve, reject) => {
   450вҶ’      // Start heartbeat monitor

   579вҶ’The Constraint Aggregation Layer ensures consistency between Strategic and Tactical planning by computing capacity envelopes.
   580вҶ’
   581вҶ’### 2.5.1 Purpose
   582вҶ’
   583вҶ’The Strategic Planner operates at quarterly granularity and cannot evaluate detailed daily constraints. However, it must not create plans that the Tactical Scheduler will reject. The Constraint Aggregation Layer bridges this gap.
   584вҶ’
   585вҶ’### 2.5.2 Aggregation Process
   586вҶ’
   587вҶ’```typescript
   588вҶ’interface ConstraintEnvelope {
   589вҶ’  period: {
   590вҶ’    start: Date;  // Quarter start
   591вҶ’    end: Date;    // Quarter end
   592вҶ’  };

   594вҶ’  // Track capacity (aggregated from daily track availability)
   595вҶ’  track_capacity: {
   596вҶ’    depot_id: string;

   598вҶ’    total_slot_days: number;      // Sum of available days in quarter
   599вҶ’    reserved_slot_days: number;   // Already committed
   600вҶ’    available_slot_days: number;  // Remaining capacity

   602вҶ’  
   603вҶ’  // Labor capacity (aggregated from weekly man-hour projections)
   604вҶ’  labor_capacity: {
   605вҶ’    depot_id: string;
   606вҶ’    regular_hours: number;        // Total regular hours in quarter
   607вҶ’    overtime_hours: number;       // Total overtime available
   608вҶ’    committed_hours: number;      // Already scheduled
   609вҶ’    available_hours: number;      // Remaining capacity
   610вҶ’  }[];
   611вҶ’  
   612вҶ’  // Material capacity (aggregated from inventory projections)
   613вҶ’  material_capacity: {
   614вҶ’    depot_id: string;

   616вҶ’    expected_available: number;   // Projected inventory
   617вҶ’    committed: number;            // Reserved for scheduled PMs
   618вҶ’    available: number;            // Remaining for new scheduling
   619вҶ’  }[];
   620вҶ’  
   621вҶ’  // Constraints from tactical model
   622вҶ’  blocked_trains: {
   623вҶ’    train_id: string;

   706вҶ’## 2.6 Key Design Principles
   707вҶ’
   708вҶ’### 2.6.1 Explainability First
   709вҶ’
   710вҶ’Every optimization decision must be traceable to specific constraints:
   711вҶ’
   712вҶ’- **Run assignments**: Include explanation factors in API response
   713вҶ’- **PM scheduling**: Log which constraints drove date selection
   714вҶ’- **Man-hour adjustments**: Record rationale for each proposed change
   715вҶ’- **Strategic allocations**: Document constraint envelope used
   716вҶ’
   717вҶ’### 2.6.2 Graceful Degradation
   718вҶ’
   719вҶ’The system must provide useful output even when optimal solutions are impossible:
   720вҶ’
   721вҶ’- **C++ solver timeout**: Fall back to TypeScript greedy solver
   722вҶ’- **Material unavailability**: Flag but continue with labor optimization
   723вҶ’- **Constraint conflicts**: Return least-bad solution with violation list
   724вҶ’- **Data quality issues**: Proceed with validated subset, quarantine bad records
   725вҶ’
   726вҶ’### 2.6.3 Audit Everything
   727вҶ’
   728вҶ’For 12-year government contract compliance:
   729вҶ’
   730вҶ’- **All schedule changes**: Who, when, what, why
   731вҶ’- **All baseline publications**: Snapshot contents, approval chain
   732вҶ’- **All scenario promotions**: Delta applied, approval
   733вҶ’- **All system decisions**: Optimizer inputs/outputs preserved
   734вҶ’
   735вҶ’### 2.6.4 Stable Long-Term Planning
   736вҶ’
   737вҶ’Prevent "jittery" forecasts that erode planner trust:
   738вҶ’
   739вҶ’- **Published baselines**: Immutable reference points
   740вҶ’- **Drift thresholds**: Only re-plan when variance exceeds 5%
   741вҶ’- **Dampening**: Strategic forecasts use rolling averages, not daily deltas
   742вҶ’- **Explicit triggers**: Users control when strategic re-planning occurs

    10вҶ’## 1. Business Context
    11вҶ’
    12вҶ’### 1.1 The Opportunity
    13вҶ’
    14вҶ’This project is a **response to the RFQ** (Request for Quotation) from a railway operator managing **128 trains across two fleets** (49 Kinki Sharyo + 79 Alstom) over a **12-year contract period**.
    15вҶ’
    16вҶ’**Current Client Pain Points:**
    17вҶ’- No dedicated mileage management tools (using Excel manually)
    18вҶ’- IBM Maximo serves only as asset management, not scheduling
    19вҶ’- Limited integration between maintenance and operations
    20вҶ’- Risk of non-compliance due to manual mileage tracking
    21вҶ’- No predictive simulations or "What-If" analysis capability
    22вҶ’
    23вҶ’### 1.2 The Objective
    24вҶ’
    25вҶ’Build an **impressive, production-ready demo** that demonstrates our capability to deliver a "Single Pane of Glass" solution for maintenance planning. The demo must:
    26вҶ’
    27вҶ’1. **Recover client confidence** after a suboptimal initial presentation
    28вҶ’2. **Prove technical excellence** through fluid, zero-latency interactions
    29вҶ’3. **Address all RFQ requirements** visually and functionally
    30вҶ’4. **Differentiate from competitors** with the "Fractal Canvas" concept
    31вҶ’
    32вҶ’### 1.3 The Core Metaphor: "Google Maps of Time"
    33вҶ’
    34вҶ’Just as Google Maps allows seamless navigation from planet view to street level, the Fractal Planner enables:
    35вҶ’
    36вҶ’- **Zoom out** вҶ’ See 12 years of strategic planning (heatmaps, capacity)
    37вҶ’- **Zoom in** вҶ’ See daily operations (Gantt, individual trains)
    38вҶ’- **Continuous transition** вҶ’ No page reloads, no mode switches
    39вҶ’- **Adaptive visualization** вҶ’ The chart type changes based on zoom level
    40вҶ’- **Bidirectional propagation** вҶ’ Changes at any level ripple up and down
    41вҶ’

    44вҶ’## 2. The Fractal Architecture
    45вҶ’
    46вҶ’### 2.1 Zoom Levels & Analysis Hierarchy
    47вҶ’
    48вҶ’The system supports **continuous zoom** with the following semantic levels:
    49вҶ’
    50вҶ’| Temporal Scale | Visible Unit | Default Visualization | Analysis Focus | Editable Actions |
    51вҶ’|----------------|--------------|----------------------|----------------|------------------|
    52вҶ’| **12 Years** | Years | Heatmap (saturation) | Total capacity, CAPEX/OPEX, overhaul cycles | Move overhauls between years |
    53вҶ’| **3-5 Years** | Semesters | Stacked Bars + Trends | Fleet lifecycle patterns | Long-term maintenance strategy |
    54вҶ’| **1-2 Years** | Quarters | Combo Chart (workload + capacity) | Seasonality, predicted peaks | Quarterly budget allocation |
    55вҶ’| **Quadrimester** | Months | Workload Equalizer | Workload vs capacity | Macro leveling |
    56вҶ’| **1-3 Months** | Weeks | Weekly Bars | Resource conflicts, event impact | Task redistribution |
    57вҶ’| **2-4 Weeks** | Days (grouped) | Simplified Gantt | Track occupancy, bottlenecks | Block movement |
    58вҶ’| **1 Week** | Days | Detailed Gantt | Per-train status, mileage | Fine scheduling, routes |
    59вҶ’| **1-3 Days** | Hours (if needed) | Capsules + Details | Fitness, remaining km | Specific route assignment |

    62вҶ’
    63вҶ’**Changes propagate in BOTH directions:**
    64вҶ’
    65вҶ’```
    66вҶ’OPERATIONAL CHANGE (Day 15/03/2026):
    67вҶ’  в””в”Җв”Җ Swap Train 104 from "Long Route" to "Short Route"
    68вҶ’       в””в”Җв”Җ Daily mileage changes: -450km
    69вҶ’            в””в”Җв”Җ Next PM prediction shifts: +3 days
    70вҶ’                 в””в”Җв”Җ March workload changes
    71вҶ’                      в””в”Җв”Җ Q1/2026 workload changes
    72вҶ’                           в””в”Җв”Җ 2026 annual cost changes
    73вҶ’                                в””в”Җв”Җ 12-year projection updates
    74вҶ’```
    75вҶ’
    76вҶ’```
    77вҶ’STRATEGIC CHANGE (12-Year View):
    78вҶ’  в””в”Җв”Җ Move Alstom Fleet Overhaul from 2027 вҶ’ 2028
    79вҶ’       в””в”Җв”Җ Budget 2027 decreases, Budget 2028 increases
    80вҶ’            в””в”Җв”Җ Monthly workload redistributes
    81вҶ’                 в””в”Җв”Җ Weekly track availability changes
    82вҶ’                      в””в”Җв”Җ Daily scheduling needs recalculation
    83вҶ’                           в””в”Җв”Җ Individual train routes may conflict
    84вҶ’```
    85вҶ’
    86вҶ’This is the "Fractal" nature: **each level is a projection of the same dataset**, and modifications cascade through all levels.

   170вҶ’## 4. Technical Architecture
   171вҶ’
   172вҶ’### 4.1 Engineering Principles
   173вҶ’
   174вҶ’- **Separation of Concerns (Transient vs. Reactive):** React manages component tree and accessibility. PixiJS manages data visualization. Animation state (drag-and-drop, zoom, pan) is **Transient** (lives outside React render cycle) to guarantee 60fps.
   175вҶ’
   176вҶ’- **Virtual Infinite Scrolling:** We abandon native browser scroll for the temporal axis. Due to DOM coordinate precision limits (~33M pixels), we implement a **Virtual Scroller** based on `wheel`/`touch` events and mathematical inertia.
   177вҶ’
   178вҶ’- **Data-Oriented Design (DoD):** To avoid Garbage Collector pressure, we minimize object allocation. We use **Structure of Arrays (SoA)** and `TypedArrays` for the simulation core, ensuring _Data Locality_ and CPU cache hits.
   179вҶ’
   180вҶ’- **Heuristics over Perfection:** For real-time interactivity (<16ms), we use Greedy algorithms and simplified Simulated Annealing. Perfect optimization (MIP) is reserved for async background processing.
   181вҶ’
   182вҶ’### 4.2 Technology Stack
   183вҶ’
   184вҶ’#### Core Frontend
   185вҶ’- **Framework:** `React 19` (SPA Mode)
   186вҶ’- **Build Tool:** `Vite`
   187вҶ’- **Language:** `TypeScript 5.x` (Strict Mode)
   188вҶ’- **State Management (UI/Server):** `Zustand` (Global UI) & `TanStack Query v5` (Async Data)
   189вҶ’- **State Management (Transient/Canvas):** `Zustand` (Vanilla Store) with direct subscriptions to avoid React re-renders during animations
   190вҶ’
   191вҶ’#### High-Performance Rendering (The Canvas)
   192вҶ’- **Engine:** `PixiJS (v8)`
   193вҶ’- **Shader Strategy:** Custom Fragment Shaders for Heatmaps (LOD 0) and Grids, moving load from CPU to GPU
   194вҶ’- **Culling:** Aggressive View Frustum Culling using Interval Trees
   195вҶ’
   196вҶ’#### Data Layer & Backend (Simulation)
   197вҶ’- **Architecture:** Robust Mock Adapter designed for future replacement by BFF/ODS
   198вҶ’- **Determinism:** Data generator must accept a **Seed** to ensure bug scenarios are reproducible (no raw `Math.random()`)
   199вҶ’
   200вҶ’### 4.3 Coordinate System & Transformations
   201вҶ’
   202вҶ’Conversion between Time (t) and Pixels (x) must be linear and high-precision (Float64):
   203вҶ’
   204вҶ’```
   205вҶ’x_viewport = (t_unix - t_origin) * S + x_offset
   206вҶ’```
   207вҶ’
   208вҶ’Where `S` is the scale (px/ms) and `x_offset` is the virtual translation.
   209вҶ’
   210вҶ’### 4.4 LOD Hysteresis
   211вҶ’
   212вҶ’To avoid flickering at level boundaries, we implement **Hysteresis**:
   213вҶ’
   214вҶ’| Level | Entry Scale | Exit Scale | Visualization | Render Technique |
   215вҶ’|-------|-------------|------------|---------------|------------------|
   216вҶ’| **LOD 0** | < 0.1 px/h | > 0.15 px/h | Heatmaps | Fragment Shader (GPU) |
   217вҶ’| **LOD 1** | 0.1 - 20 px/h | < 0.08 or > 25 px/h | Stacked Bars | Instanced Mesh (PixiJS) |
   218вҶ’| **LOD 2** | > 20 px/h | < 15 px/h | Gantt / Capsules | Graphics / Sprites |
   219вҶ’
   220вҶ’> The overlapping zones (e.g., entry at 0.1, exit at 0.15) ensure visual stability.
   221вҶ’
   222вҶ’### 4.5 Navigation Strategy (Virtual Scroller)
   223вҶ’
   224вҶ’Instead of a "Ghost Div":
   225вҶ’
   226вҶ’1. **Input Capture:** A transparent container captures `wheel`, `pointerdown`, `pointermove` events
   227вҶ’2. **Physics Engine:** An animation loop calculates velocity and friction (inertia/momentum)
   228вҶ’3. **State Update:** Updates `viewport.x` in the transient store
   229вҶ’4. **Render Trigger:** PixiJS draws the frame based on new `viewport.x`
   230вҶ’5. **Synchronization:** React is notified only when movement stops (idle) or every X ms (throttled) to update the DOM time ruler

   234вҶ’## 5. The Constraint Engine (Client-Side Simulation)
   235вҶ’
   236вҶ’### 5.1 Data Structure (SoA - Structure of Arrays)
   237вҶ’
   238вҶ’To support 50,000+ blocks without GC "hiccups":
   239вҶ’
   240вҶ’```typescript
   241вҶ’// Conceptual SoA example
   242вҶ’class SimulationStore {
   243вҶ’  // Pre-allocated capacity
   244вҶ’  capacity = 100000;
   245вҶ’
   246вҶ’  // TypedArrays: Contiguous, fast memory access
   247вҶ’  startTimes = new Float64Array(this.capacity);
   248вҶ’  endTimes = new Float64Array(this.capacity);
   249вҶ’  trainIds = new Int32Array(this.capacity); // Mapped to string IDs
   250вҶ’  types = new Uint8Array(this.capacity);    // Enum: 0=PM, 1=Overhaul...
   251вҶ’
   252вҶ’  // Spatial Index for fast queries
   253вҶ’  intervalTree = new FlatIntervalTree(this.startTimes, this.endTimes);
   254вҶ’}
   255вҶ’```
   256вҶ’
   257вҶ’### 5.2 Game Loop & Validation
   258вҶ’
   259вҶ’1. **Drag Start:** Identify block index in Arrays
   260вҶ’2. **Move (Tick):**
   261вҶ’   - Calculate provisional new `startTime`
   262вҶ’   - **Collision Detection:** Query `IntervalTree` for conflicts on same `trainId` or `trackId`
   263вҶ’   - **Visual Feedback:** Render block at new position (green/red). **Do NOT update React DOM.**
   264вҶ’3. **Drop (Commit):**
   265вҶ’   - Apply change to TypedArrays
   266вҶ’   - Trigger sync with Mock API
   267вҶ’   - Update aggregated metrics

   318вҶ’## 9. Success Criteria

     1‚Üí# Metro Integrated Planning System (MMS)
     2‚Üí
     3‚Üí## Project Overview
     4‚ÜíHigh-Performance Enterprise Integrated Planning System with "Semantic Zoom" UX (Google Earth-style) for metro fleet management. Renders 50+ trains and 10 years of tasks at constant 60fps using WebGL.
     5‚Üí

     8‚Üí## HARNESS: Fundamental Concept (READ THIS FIRST)
     9‚Üí
    10‚Üí> **This project uses a Harness.** If you don't know what that is, STOP and research:
    11‚Üí> - [Effective harnesses for long-running agents](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents)
    12‚Üí
    13‚Üí### What is a Harness?
    14‚Üí
    15‚ÜíA **Harness** is a framework that manages agent execution across multiple sessions. Each new session starts with NO memory‚Äîharnesses bridge this gap.
    16‚Üí
    17‚Üí> Source: [Effective harnesses for long-running agents](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents)
    18‚Üí
    19‚Üí**Key Insight from Anthropic:**
    20‚Üí> "The core challenge is that agents must work in discrete sessions, and each new session begins with no memory. Agents need a way to quickly understand the state of work when starting with a fresh context window."
    21‚Üí
    22‚Üí**Solution:** State files persist context:
    23‚Üí- `claude-progress.txt` ‚Üí session history (what was done)
    24‚Üí- `features.json` ‚Üí requirements with status (JSON preferred‚Äîless likely to be overwritten)
    25‚Üí
    26‚Üí**Failure Patterns to Avoid:**
    27‚Üí1. **Doing too much:** Running out of context mid-task, unable to pass clear instructions to next session
    28‚Üí2. **Declaring done prematurely:** Seeing progress and stopping before actually completing
    29‚Üí
    30‚Üí### Work Protocol
    31‚Üí
    32‚Üí```
    33‚Üí1. READ claude-progress.txt (understand current state)
    34‚Üí2. READ features.json (find next feature)
    35‚Üí3. IMPLEMENT one feature
    36‚Üí4. TEST visually (Playwright MCP)
    37‚Üí5. COMMIT after passing
    38‚Üí6. UPDATE claude-progress.txt
    39‚Üí7. STOP (leave codebase production-ready)
    40‚Üí```
    41‚Üí
    42‚Üí### üî¥ CRITICAL: Incremental Saving Protocol
    43‚Üí
    44‚Üí> **LESSON LEARNED (2025-12-20):** Previous sessions lost 91% of context because
    45‚Üí> they only saved to harness at the end. If context overflows, everything is lost.
    46‚Üí
    47‚Üí**Context Limit:** ~168,000 tokens per session. At 4% remaining, you only have ~6,700 tokens left.
    48‚Üí
    49‚Üí**Automated Enforcement:**
    50‚Üí
    51‚ÜíThe harness has hooks that will **warn you** when checkpoints are needed:
    52‚Üí- `UserPromptSubmit` ‚Üí `harness-checkpoint.sh` (warns if progress > 15 min stale)
    53‚Üí- `Stop` ‚Üí `end-of-turn-check.sh` (warns if progress > 10 min stale + uncommitted changes)
    54‚Üí
    55‚Üí**Mandatory Checkpoints:**
    56‚Üí
    57‚Üí| When                     | Action                    |
    58‚Üí|--------------------------|---------------------------|
    59‚Üí| BEFORE starting any task | Register in harness       |
    60‚Üí| AFTER each subtask       | Update harness            |
    61‚Üí| Every 10 minutes max     | Checkpoint save           |
    62‚Üí| BEFORE intensive ops     | Save current state        |
    63‚Üí| If context > 70% full    | STOP and save IMMEDIATELY |
    64‚Üí
    65‚Üí**Ask Yourself:**
    66‚Üí
    67‚Üí> "If this chat were lost RIGHT NOW, would the next agent have
    68‚Üí> enough context to continue without loss?"
    69‚Üí
    70‚ÜíIf the answer is NO ‚Üí **SAVE NOW** (update harness + git commit).
    71‚Üí
    72‚Üí**Baby Steps Commits:** Commit small, frequent changes. A WIP commit is better than lost work.

    92‚Üí## Parallel Execution (Git Worktrees)
    93‚Üí
    94‚ÜíThis project supports **parallel feature development** using git worktrees. This enables multiple Claude sessions to work on different features simultaneously, achieving ~2.5x faster development.
    95‚Üí
    96‚Üí### When to Use Parallel Execution
    97‚Üí
    98‚ÜíUse parallel execution when:
    99‚Üí- Multiple independent features can be worked on concurrently
   100‚Üí- Features modify different files (low conflict risk)
   101‚Üí- You want to maximize development throughput
   102‚Üí
   103‚Üí### Quick Start
   104‚Üí
   105‚Üí```bash
   106‚Üí# Show current status of all waves/features
   107‚Üíbun scripts/parallel-runner.ts status
   108‚Üí
   109‚Üí# Execute Wave 1 (starts 3 parallel groups)
   110‚Üíbun scripts/parallel-runner.ts wave 1
   111‚Üí
   112‚Üí# Merge a completed feature back to main
   113‚Üíbun scripts/parallel-runner.ts merge F023
   114‚Üí
   115‚Üí# Clean up all worktrees
   116‚Üíbun scripts/parallel-runner.ts cleanup
   117‚Üí```
   118‚Üí
   119‚Üí### Wave Structure
   120‚Üí
   121‚ÜíFeatures are organized into 4 waves based on dependencies and file conflicts:
   122‚Üí
   123‚Üí```
   124‚ÜíWAVE 1 (No dependencies - 3 parallel groups):
   125‚Üí‚îú‚îÄ‚îÄ Group A: F023 (Dependency lines) ‚Üí MicroLayer.tsx
   126‚Üí‚îú‚îÄ‚îÄ Group B: F123, F025 (Minimap) ‚Üí App.tsx [sequential]
   127‚Üí‚îî‚îÄ‚îÄ Group C: F016, F017 (Scroll modifiers) ‚Üí InfiniteCanvas.tsx [sequential]
   128‚Üí
   129‚ÜíWAVE 2 (After Wave 1 - 2 parallel groups):
   130‚Üí‚îú‚îÄ‚îÄ Group D: F027, F028 (Keyboard shortcuts) ‚Üí InfiniteCanvas.tsx [sequential]
   131‚Üí‚îî‚îÄ‚îÄ Group E: F026 (Task details panel) ‚Üí App.tsx
   132‚Üí
   133‚ÜíWAVE 3 (After Wave 2 - 1 group):
   134‚Üí‚îî‚îÄ‚îÄ Group F: F021 (Keyboard navigation)
   135‚Üí
   136‚ÜíWAVE 4 (After Wave 3 - 1 group):
   137‚Üí‚îî‚îÄ‚îÄ Group G: F022 (ARIA labels)
   138‚Üí```
   139‚Üí
   140‚Üí### How It Works
   141‚Üí
   142‚Üí1. **Worktrees**: Each feature gets its own directory (`.trees/<feature-id>/`) with a dedicated branch
   143‚Üí2. **Isolation**: Each Claude session only modifies files for its assigned feature
   144‚Üí3. **Merging**: After a feature is complete, it's merged back to `main`
   145‚Üí4. **Dependencies**: Waves are executed in order; a wave can't start until the previous completes
   146‚Üí
   147‚Üí### Key Files
   148‚Üí
   149‚Üí| File | Purpose |
   150‚Üí|------|---------|
   151‚Üí| `scripts/parallel-runner.ts` | Interactive runner with worktree management |
   152‚Üí| `scripts/parallel-config.ts` | Wave/group definitions and conflict analysis |
   153‚Üí| `docs/PARALLEL_EXECUTION.md` | Complete documentation |
   154‚Üí| `.trees/` | Directory for git worktrees (gitignored) |
   155‚Üí
   156‚Üí### Conflict Risk Levels
   157‚Üí
   158‚Üí- **LOW**: Features modify isolated files (e.g., only MicroLayer.tsx)
   159‚Üí- **MEDIUM**: Features modify shared files but different sections (e.g., App.tsx layout)
   160‚Üí- **HIGH**: Features modify the same functions (e.g., handleWheel in InfiniteCanvas.tsx)
   161‚Üí
   162‚ÜíSequential groups (marked `[sequential]`) contain features that must run one after another due to HIGH conflict risk.
   163‚Üí
   164‚Üí**Full documentation:** See `docs/PARALLEL_EXECUTION.md`

   206‚Üí## KISS Philosophy
   207‚Üí
   208‚Üí**KISS is NOT "do the minimum". It IS "don't add complexity that blocks progress".**
   209‚Üí
   210‚ÜíThe goal is **speed of delivery with quality**. The product must impress the client to win contracts.
   211‚Üí
   212‚Üí### What KISS means (examples)
   213‚Üí
   214‚Üí| ‚úÖ DO                                                         | ‚ùå DON'T                                                          |
   215‚Üí|--------------------------------------------------------------|------------------------------------------------------------------|
   216‚Üí| Add polish, debug tools, UX refinements that accelerate work | Spend days on infrastructure before having something presentable |
   217‚Üí| Use best practices that prevent future problems              | Add abstractions for hypothetical future needs                   |
   218‚Üí| Preserve knowledge/tips for when they become relevant        | Ignore proven patterns just because "not needed yet"             |
   219‚Üí
   220‚Üí### When to Increase Complexity (examples)
   221‚Üí
   222‚Üí| ‚úÖ Increase complexity when...     | ‚ùå Do NOT increase when... |
   223‚Üí|-----------------------------------|---------------------------|
   224‚Üí| Current approach blocks a feature | "It would be cleaner"     |
   225‚Üí| Performance is unacceptable       | "Best practices say so"   |
   226‚Üí| Code duplication becomes painful  | Premature optimization    |
   227‚Üí| Team needs shared abstractions    | Speculative future needs  |
   228‚Üí
   229‚Üí### Example: POC-first approach
   230‚Üí

   338‚Üí### State Management: AVOID React Hooks, USE @mms/toolkit
   339‚Üí
   340‚Üí> **CRITICAL RULE:** Avoid `useState`/`useRef` for state. Use `@mms/toolkit` state-machine for ALL state.
   341‚Üí
   342‚Üí**Why?**
   343‚Üí- React hooks cause "stale closures" in animation loops
   344‚Üí- State updates are batched/async - values become stale
   345‚Üí- Debugging is hard when state is scattered across multiple hooks
   346‚Üí- @mms/toolkit uses `useSyncExternalStore` - always synchronous and consistent
   347‚Üí
   348‚Üí**Current architecture (after TECH-DEBT-001):**
   349‚Üí```
   350‚Üíapps/canvas/src/state/
   351‚Üí‚îú‚îÄ‚îÄ canvas-machine.ts   # Main machine
   352‚Üí‚îú‚îÄ‚îÄ viewport.ts         # Slice: camera, zoom, momentum, animation
   353‚Üí‚îú‚îÄ‚îÄ interaction.ts      # Slice: hover, selection, mouse position
   354‚Üí‚îî‚îÄ‚îÄ index.ts            # Re-exports
   355‚Üí```
   356‚Üí
   357‚Üí**How to use:**
   358‚Üí```typescript
   359‚Üí// ‚úÖ CORRECT: Use state-machine
   360‚Üíimport { useCanvasSelector, canvasStore, canvasActions } from '../state';
   361‚Üí
   362‚Üí// Read state (reactive - causes re-render on change)
   363‚Üíconst zoom = useCanvasSelector(s => s.viewport.camera.zoom);
   364‚Üí
   365‚Üí// Read state (synchronous - for animation loops)
   366‚Üíconst camera = canvasStore.getState().viewport.camera;
   367‚Üí
   368‚Üí// Update state
   369‚ÜícanvasStore.dispatch(canvasActions.viewport.setCamera({ x, y, zoom }));
   370‚Üí```
   371‚Üí
   372‚Üí```typescript
   373‚Üí// ‚ùå AVOID: React hooks for state
   374‚Üíconst [zoom, setZoom] = useState(1); // NO - stale closures
   375‚Üíconst zoomRef = useRef(1);           // NO - unnecessary workaround
   376‚Üí```
   377‚Üí
   378‚Üí**Exceptions (use hooks only for):**
   379‚Üí- `useCallback`/`useMemo` for render optimization
   380‚Üí- `useEffect` for side effects (event listeners, cleanup)
   381‚Üí- `useRef` ONLY for DOM references (`containerRef`)
   382‚Üí- 100% local disposable UI state (e.g., temporary tooltip position)
   383‚Üí
   384‚Üí### High-Performance Architecture (STATE-ARCH-001)
   385‚Üí
   386‚Üí> **Research:** See `docs/state-management-research/relatorio-state-management.pdf`
   387‚Üí
   388‚ÜíThe research document identifies **Garbage Collection as the primary bottleneck** in WebGL applications, not rendering. Traditional React patterns (useState, Proxy-based stores) are inadequate for 60fps updates on mobile.
   389‚Üí
   390‚Üí**Planned Dual-Store Architecture:**
   391‚Üí
   392‚Üí| Aspect | Simulation State | Application State |
   393‚Üí|--------|-----------------|-------------------|
   394‚Üí| Purpose | Entity data (60fps) | UI state (event-driven) |
   395‚Üí| Technology | TypedArrays / SoA pattern | **@mms/toolkit** (NEVER Zustand) |
   396‚Üí| Mutability | Direct (performance) | Immutable (Immer) |
   397‚Üí| Access | Indices / getState() | useSelector / subscriptions |
   398‚Üí
   399‚Üí**Key Principles from Research:**
   400‚Üí
   401‚Üí1. **Structure of Arrays (SoA):** Store entity properties in separate TypedArrays for cache-friendly iteration
   402‚Üí2. **Object Pooling:** Pre-allocate vectors/objects to eliminate GC pauses in animation loops
   403‚Üí3. **Spatial Indexing:** Use RBush (R-tree) for O(log n) viewport queries
   404‚Üí4. **UI Throttling:** Sync Engine‚ÜíUI at 30fps max, not every frame
   405‚Üí
   406‚Üí**CRITICAL: Library Choice**
   407‚Üí
   408‚Üí```
   409‚Üí‚úÖ UI State:     @mms/toolkit (packages/toolkit)
   410‚Üí‚úÖ Entity Data:  Custom TypedArray store (to be implemented)
   411‚Üí‚úÖ Undo/Redo:    Command Pattern + Immer patches
   412‚Üí
   413‚Üí‚ùå NEVER USE:    Zustand, Redux, MobX, Valtio
   414‚Üí                 (despite PDF recommendations, we use @mms/toolkit)
   415‚Üí```
   416‚Üí
   417‚Üí**Implementation:** See `features.json` ‚Üí `STATE-ARCH-001` for phased plan.

   560‚Üí## MCP VISUAL TESTING (MANDATORY)
   561‚Üí
   562‚Üí### Why This is Required
   563‚Üí
   564‚ÜíYou cannot see the app you're building. Without visual testing, you will:
   565‚Üí- Mark blank screens as "working"
   566‚Üí- Miss CSS/layout issues
   567‚Üí- Overlook runtime errors visible only in UI
   568‚Üí
   569‚Üí### Testing Protocol: Playwright MCP
   570‚Üí
   571‚ÜíUse **Playwright MCP** for visual tests. Runs headless and **does not steal focus** from the user's window.
   572‚Üí
   573‚Üí#### Step 1: Navigate to the page
   574‚Üí
   575‚Üí```
   576‚Üímcp__playwright__browser_navigate url="file:///full/path/to/file.html"
   577‚Üí```
   578‚Üí
   579‚ÜíOr for local server:
   580‚Üí```
   581‚Üímcp__playwright__browser_navigate url="http://localhost:5173"
   582‚Üí```
   583‚Üí
   584‚Üí#### Step 2: Check console (JS errors)
   585‚Üí
   586‚Üí```
   587‚Üímcp__playwright__browser_console_messages level="error"
   588‚Üí```
   589‚Üí
   590‚Üí- Red errors ‚Üí **FAIL** (investigate before continuing)
   591‚Üí- Yellow warnings ‚Üí Evaluate (e.g., "Tailwind CDN production warning" is OK)
   592‚Üí
   593‚Üí#### Step 3: Capture screenshot
   594‚Üí
   595‚Üí```
   596‚Üímcp__playwright__browser_take_screenshot filename="screenshots/test-name.png"
   597‚Üí```
   598‚Üí
   599‚ÜíFor scrollable pages:
   600‚Üí```
   601‚Üímcp__playwright__browser_take_screenshot fullPage=true filename="screenshots/test-full.png"
   602‚Üí```
   603‚Üí
   604‚Üí#### Step 4: Analyze visually
   605‚Üí
   606‚ÜíCheck in the screenshot:
   607‚Üí- [ ] Content visible? (not blank/empty)
   608‚Üí- [ ] Layout correct? (elements positioned as expected)
   609‚Üí- [ ] Colors correct? (matching design)
   610‚Üí- [ ] No visual artifacts?
   611‚Üí
   612‚Üí#### Step 5: Interactions (if needed)
   613‚Üí
   614‚Üí```
   615‚Üímcp__playwright__browser_click element="button X" ref="e123"
   616‚Üímcp__playwright__browser_type element="field Y" ref="e456" text="value"
   617‚Üí```
   618‚Üí
   619‚Üí#### Step 6: Close browser at the end
   620‚Üí
   621‚Üí```
   622‚Üímcp__playwright__browser_close
   623‚Üí```

   626‚Üí
   627‚Üí| Criterion | PASS | FAIL |

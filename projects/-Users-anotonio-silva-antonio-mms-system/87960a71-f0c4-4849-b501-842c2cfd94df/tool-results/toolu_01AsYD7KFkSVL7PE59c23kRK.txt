    27→## Executive Summary
    28→
    29→Railway rolling stock maintenance scheduling is a complex optimization problem that combines:
    30→- **Resource allocation** (man-hours, tracks, depots)
    31→- **Time/mileage constraints** (preventive maintenance intervals)
    32→- **Operational requirements** (train availability, service levels)
    33→- **Cost optimization** (labor, materials, penalties)
    34→
    35→Key findings from our research:
    36→
    37→| Aspect | Industry Standard |
    38→|--------|------------------|
    39→| Planning horizons | Daily → Weekly → Monthly → Annual → Long-term (10+ years) |
    40→| Primary trigger | Mileage-based (evolving to condition-based) |
    41→| Common techniques | MIP, Genetic Algorithms, Large Neighborhood Search, Hybrid CP/MIP |
    42→| Software platforms | IBM Maximo, SAP EAM, Custom optimization engines |
    43→| Performance target | 99.9% train availability (Hong Kong MTR benchmark) |
    44→

    47→## Problem Definition
    48→
    49→### The Core Scheduling Problem
    50→
    51→The railway maintenance scheduling problem involves:
    52→
    53→1. **Fleet Management**: Multiple train types with different maintenance requirements
    54→2. **Maintenance Types**: Varying intervals and durations
    55→   - Daily inspections
    56→   - Monthly checks (~15,000 km)
    57→   - Quarterly services (~45,000 km)
    58→   - Annual overhauls (~180,000 km)
    59→   - Major overhauls (4-6 years)
    60→
    61→3. **Resource Constraints**:
    62→   - Limited maintenance tracks
    63→   - Fixed workforce (man-hours per period)
    64→   - Depot capacity
    65→   - Spare parts availability
    66→
    67→4. **Objectives** (often conflicting):
    68→   - Minimize maintenance costs
    69→   - Maximize train availability
    70→   - Level resource utilization
    71→   - Comply with safety regulations
    72→
    73→### Problem Complexity
    74→
    75→According to research from [Wiley - Optimization of Annual Planned Rail Maintenance](https://onlinelibrary.wiley.com/doi/10.1111/mice.12764):
    76→
    77→> "Using the CPLEX MIP solver, researchers managed to solve only 62% of instances within 3 hours. Kiefer et al. (2018) formulated their problem as an MIP model but quickly found runtime limitations on larger instances, exceeding the 24-hour time limit."
    78→
    79→This demonstrates why heuristic and metaheuristic approaches are essential for real-world applications.

   126→## Optimization Techniques
   127→
   128→### 1. Mixed Integer Programming (MIP)
   129→
   130→**Strengths:**
   131→- Mathematically optimal solutions
   132→- Well-understood theory
   133→- Commercial solvers available (Gurobi, CPLEX)
   134→
   135→**Limitations:**
   136→- Scalability issues with large instances
   137→- Long computation times
   138→- Requires linearization of constraints
   139→
   140→**Example Application - Deutsche Bahn:**
   141→
   142→From [Deutsche Bahn Hypergraph Optimization Case Study](https://pubsonline.informs.org/doi/10.1287/inte.2020.1069):
   143→
   144→> "DB attributes annual savings of €74 million, an annual reduction of 34,000 tons of CO2 emissions, and the elimination of 600 coupling operations to the implementation of FEO (Fleet Employment Optimization)."
   145→
   146→### 2. Genetic Algorithms (GA)
   147→
   148→**Strengths:**
   149→- Handles complex, non-linear constraints
   150→- Good for multi-objective optimization
   151→- Scalable to large problems
   152→
   153→**Example Application - Hong Kong MTR:**
   154→
   155→From [CityU AI Projects - MTR Scheduling](https://www.cs.cityu.edu.hk/~hwchun/AIProjects/stories/allocation/mtrcscheduling/):
   156→
   157→> "The scheduling system is based on genetic algorithm and heuristic search, which work together with a rule-based system. The AI system improved maintenance efficiency by 50%, contributing to MTR's 99.9% on-time performance for 5 million daily passengers."
   158→
   159→### 3. Large Neighborhood Search (LNS)
   160→
   161→**Strengths:**
   162→- Effective for large-scale problems
   163→- Combines local search with diversification
   164→- Can escape local optima
   165→
   166→**Example Application - Vienna Tram:**
   167→
   168→From [ScienceDirect - Integrated Railway Planning](https://www.sciencedirect.com/science/article/abs/pii/S0968090X16302340):
   169→
   170→> "Kiefer et al. developed a metaheuristic based on large neighborhood search after MIP exceeded 24-hour time limits. Testing on the Vienna tram network—the largest instance in literature—showed effective results."
   171→
   172→### 4. Hybrid Constraint Programming / MIP
   173→
   174→**Strengths:**
   175→- CP generates feasible solutions quickly
   176→- MIP improves solution quality
   177→- Best of both worlds
   178→
   179→**Example Application - Danish Railways:**
   180→
   181→From [ScienceDirect - Hybrid CP/MIP Framework](https://www.sciencedirect.com/science/article/pii/S0377221717307646):
   182→
   183→> "Whereas using a general purpose MIP solver is only able to solve instances over a two-week planning horizon, the hybrid framework generates good results for problem instances over an eight-week period."
   184→
   185→### 5. Greedy Metaheuristics
   186→
   187→**Strengths:**
   188→- Fast computation
   189→- Good initial solutions
   190→- Easy to implement
   191→
   192→**Example Application - Netherlands ProRail:**
   193→
   194→From [Wiley - Annual Rail Maintenance Optimization](https://onlinelibrary.wiley.com/doi/10.1111/mice.12764):
   195→
   196→> "A hybrid algorithm using evolution strategy and a greedy algorithm was created and shown to outperform all other algorithms, achieving a two-thirds decrease in costs compared to human scheduler solutions."
   197→
   198→### Technique Comparison Matrix
   199→
   200→| Technique | Problem Size | Solution Quality | Speed | Implementation |
   201→|-----------|--------------|------------------|-------|----------------|
   202→| MIP | Small-Medium | Optimal | Slow | Complex |
   203→| Genetic Algorithm | Large | Good | Medium | Moderate |
   204→| Large Neighborhood Search | Large | Good | Fast | Moderate |
   205→| Hybrid CP/MIP | Medium-Large | Very Good | Medium | Complex |
   206→| Greedy Heuristics | Any | Acceptable | Very Fast | Simple |
   207→
   208→### Algorithm Selection Strategy for MMS
   209→
   210→Based on the research, a layered algorithmic architecture is recommended:
   211→
   212→1. **Strategic/Long-Term (Annual):** MIP for capacity planning and yearly workload leveling
   213→2. **Tactical/Medium-Term (Weeks):** CP or MIP with decomposition (LBBD) for slot assignment respecting depot constraints
   214→3. **Operational/Real-Time (Hours):** Heuristics and Metaheuristics for fast rescheduling during disruptions

   218→## Physical Constraints Modeling
   219→
   220→A common failure in traditional planning systems is over-abstraction of physical reality. The MMS must rigorously model depot and rolling stock constraints to ensure generated plans are executable.
   221→
   222→### Pit Occupancy and Finite Capacity
   223→
   224→Depot capacity is not just a track count; it's a complex resource and geometry constraint problem.
   225→
   226→**The "Tetris" Spatial Problem:**
   227→Trains have different lengths (4-car, 8-car). Pits have fixed lengths. A "Pit Occupancy" constraint must model spatial arrangement:
   228→- Can two 4-car trains fit in an 8-car pit?
   229→- What order must they enter (LIFO - Last-In-First-Out)?
   230→
   231→This requires modeling tracks not just as "resources" but as "capacitated resources" with linear spatial dimensions.
   232→
   233→**Finite Capacity Modeling:**
   234→```
   235→∑ Occupancy_t ≤ Capacity_t  (for all resource types simultaneously)
   236→```
   237→
   238→**Deadlock Prevention:**
   239→In dense depots, one train's movement can block another. "Deadlocks" occur when circular movement dependencies prevent any train from moving.
   240→
   241→The MMS must incorporate deadlock prevention algorithms (such as adaptations of the Banker's Algorithm or graph cycle detection) during shunting planning to ensure every scheduled movement sequence is geometrically viable.
   242→
   243→### Shunting Time and Dynamics
   244→
   245→The time to move a train from parking track to maintenance pit is non-zero and variable.
   246→
   247→**Train Unit Shunting Problem (TUSP):**
   248→This involves matching arriving train units with departing services and parking them in LIFO stacks on depot tracks. Stack order is critical; a train departing at 05:00 cannot be blocked by one leaving at 08:00.
   249→
   250→**Crossover Conflicts:**
   251→Modeling must account for conflicts where a shunting movement crosses a switch that blocks other movements. Constraints must impose temporal separation:
   252→```
   253→If Train A uses Switch S at time t,
   254→Train B cannot use Switch S until t + clearance_time
   255→```
   256→
   257→**Shunting Duration:**
   258→The system cannot assume instantaneous movement. Shunting times must be modeled based on track topology and speed limits (typically 5 km/h in yards).
   259→
   260→### Possession Windows and Infrastructure Availability
   261→
   262→Rolling stock maintenance often competes with infrastructure maintenance (permanent way, signaling, catenary).
   263→
   264→**Possession Windows:**
   265→Constraints must align rolling stock movements with possession windows—periods when tracks are closed for maintenance. This requires a unified data model where infrastructure availability acts as a hard constraint on fleet movement.

   506→## Data Architecture
   507→
   508→The complexity of an MMS requires a data architecture that can handle high-frequency sensor data (Digital Twin), complex transactional planning (Scheduling), and historical analysis (Reporting/BI). Research points to the superiority of the CQRS pattern over traditional CRUD architectures.
   509→
   510→### CQRS (Command Query Responsibility Segregation)
   511→
   512→Traditional CRUD architectures struggle when read and write patterns are vastly different, as in an MMS.
   513→
   514→**Write Model (Command):**
   515→The "Command" side handles the complex business logic of scheduling—constraint validation, state updates, and user override processing. This requires a highly normalized transactional database (OLTP) to ensure data integrity and ACID properties.
   516→
   517→**Read Model (Query):**
   518→The "Query" side is optimized for visualization. Planners need to see Gantt charts, depot occupancy heatmaps, and fleet status summaries instantly. CQRS allows the creation of denormalized "projections" specifically designed for these views, enabling fast load times without complex joins.
   519→
   520→**Strategic Benefit:**
   521→This separation allows the heavy computational load of optimization algorithms (running on the Write side) not to impact the responsiveness of the user interface (reading from the Read side).
   522→
   523→### Event Sourcing as Source of Truth
   524→
   525→Event Sourcing complements CQRS by storing *state changes* (events) rather than just current state.
   526→
   527→**Auditability and "Time Travel":**
   528→In a safety-critical industry like rail, knowing *who* changed a schedule and *why* is crucial. Event Sourcing stores an immutable log of events:
   529→- `MaintenanceScheduled`
   530→- `TrainDelayed`
   531→- `InspectionCompleted`
   532→- `ConstraintOverridden`
   533→
   534→This allows the system to reconstruct schedule state at any point in the past, facilitating post-incident analysis and "what-if" simulations.
   535→
   536→**Digital Twin Integration:**
   537→IoT sensor events (e.g., `WheelWearThresholdExceeded`) can be ingested directly as events into the log, triggering downstream reactions in scheduling logic.
   538→
   539→### Transactional vs. Analytical Data Stores
   540→
   541→| Aspect | Transactional (OLTP) | Analytical (OLAP) |
   542→|--------|---------------------|-------------------|
   543→| Purpose | Main scheduling engine | BI and reporting |
   544→| Consistency | Immediate consistency | Eventual consistency |
   545→| Query Pattern | Single-record lookups | Complex aggregations |
   546→| Tool Example | PostgreSQL, SQL Server | Apache Superset, Data Warehouse |
   547→| MMS Use | Plan generation, real-time dispatch | "Average pit occupancy last year", trend analysis |
   548→
   549→BI tools like Apache Superset should connect to a separate Data Warehouse (fed by Read Model projections) without blocking the operational database used by real-time dispatchers.

   704→## Risk Management & Edge Cases
   705→
   706→A plan is only as good as its resilience to reality. The MMS must account for the stochastic nature of railway operations and be prepared for the unexpected.
   707→
   708→### Risk and Edge Case Catalog
   709→
   710→| Category | Risk / Edge Case | Description & Impact | Software Mitigation Strategy |
   711→|:---------|:-----------------|:---------------------|:-----------------------------|
   712→| **Technological** | **Sensor Drift** | IoT sensors report false positives or lose calibration, leading to unnecessary maintenance | Data fusion algorithms and cross-validation with history |
   713→| **Technological** | **Data Gaps** | Loss of connectivity in tunnels results in incomplete telemetry | Data interpolation (Kriging) and Bayesian predictive models |
   714→| **Operational** | **Yard Deadlocks** | Circular blocking where no train can move, paralyzing the depot | Graph cycle detection algorithms integrated into shunting module |
   715→| **Operational** | **Cascade Delays** | A minor delay makes the maintenance plan infeasible | Robust rescheduling with delay "absorption" metrics |
   716→| **Operational** | **Infeasibility** | Scenarios where *no* solution satisfies all hard constraints | Use "Soft Constraints" with penalty minimization (offer the "least bad" option) |
   717→| **Physical** | **Component Cannibalization** | Removing parts from one train to fix another, complicating asset tracking | Serialized component tracking in data model (dynamic asset tree) |
   718→| **Physical** | **Weather Events** | Extreme heat expanding rails or leaves on track affecting braking | Dynamic adjustment of shunting time parameters and speed constraints |
   719→
   720→### Robustness and Stability Metrics
   721→
   722→**Nervousness:**
   723→A schedule is "nervous" if small input changes lead to massive plan changes. To mitigate, the MMS must employ "stability objectives" in the optimization function, penalizing alterations to already published or communicated tasks:
   724→
   725→```
   726→Minimize (Operational_Cost + λ × Instability_Cost)
   727→```
   728→
   729→**Key Robustness Metrics:**
   730→
   731→| Metric | Definition | Target |
   732→|--------|------------|--------|
   733→| Absorption | Ability of schedule to absorb small delays without rescheduling | Maximize |
   734→| Recovery Time | How quickly system returns to original plan after disruption | Minimize |
   735→| Passenger Impact | Delays weighted by affected passengers | Minimize |
   736→
   737→### Software Risk Mitigation Strategies
   738→
   739→1. **Algorithmic Risk (Solver Timeout):** Implement an "Algorithm Portfolio". System tries MIP; if time exceeds limit (e.g., 5 min), automatically falls back to Greedy Heuristic or Metaheuristic to guarantee a solution is delivered.
   740→
   741→2. **Data Integrity Risk:** Implement "Data Quality Gatekeepers" at the ingestion layer. Sensor data violating physical rules (e.g., temperature rising 1000°C in 1 second) should be flagged as anomalies and not used to trigger maintenance without human validation.
   742→
   743→3. **Adoption Risk:** Combat planner resistance with Explainable AI (XAI) features and HITL workflows that respect user authority.

   747→## Human-in-the-Loop Integration
   748→
   749→Full automation is neither viable nor desirable in complex rail environments. The MMS must be designed as a decision support system that empowers human planners, not replaces them.
   750→
   751→### Visualization and Trust
   752→
   753→Algorithms cannot be black boxes. The system must visualize *why* a conflict was flagged or *why* a specific slot was chosen.
   754→
   755→**Conflict Visualization:**
   756→Instead of just displaying an "Infeasible" error, the UI should draw a red line on the Gantt chart showing exactly where the overlap or constraint violation is.
   757→
   758→**Interactive Gantt Charts:**
   759→Planners think visually. The UI should allow rescheduling via "drag-and-drop", with the system immediately recalculating and visualizing the impact of the change (delay propagation, constraint violations) in real-time.
   760→
   761→### Collaborative Planning and Veto Power
   762→
   763→**Veto Power and "Anchor & Re-plan":**
   764→In the "Automated/Person-in-Loop" scenario, the algorithm proposes a schedule, but the human planner retains veto power. If the planner "anchors" (locks) a task to a specific time, the system must re-optimize the rest of the plan around that fixed constraint.
   765→
   766→**What-If Analysis:**
   767→Planners must be able to run hypothetical scenarios (e.g., "What if we close Pit 3 for renovation next week?") and visually compare results against the base plan, seeing side-by-side impact metrics (e.g., "+2 Delayed Tasks", "+10min Overtime").
   768→
   769→### Explanation and Transparency
   770→
   771→The system should provide clear explanations for its recommendations:
   772→
   773→| Question | System Should Explain |
   774→|----------|----------------------|
   775→| "Why this train?" | Mileage proximity to next service, availability score |
   776→| "Why this slot?" | Resource availability, precedence constraints satisfied |
   777→| "Why the conflict?" | Which specific constraint is violated, by how much |
   778→| "What's the impact?" | Downstream effects on dependent tasks |

   782→## Change Propagation & Recalculation Mechanics
   783→
   784→> This section details how the system should respond when parameters change.
   785→> Critical for understanding rescheduling behavior and implementing What-If scenarios.
   786→
   787→### The Core Question
   788→
   789→**When a parameter changes, WHAT needs to be recalculated and in WHAT ORDER?**
   790→
   791→### Time Zones and Frozen Periods
   792→
   793→The system operates with hierarchical time zones that control what can be changed:
   794→
   795→| Time Horizon | Zone Status | Constraint Type | Allowed Changes | Recalculation Agent |
   796→|--------------|-------------|-----------------|-----------------|---------------------|
   797→| T > 12 weeks | **Liquid** | Soft | Structure, Volume, Frequency | Strategic Planner (MIP) |
   798→| T-12 to T-7 weeks | **Slushy** | Medium | Timing, Resource Allocation | Tactical Planner (Hybrid) |
   799→| T-7 weeks to T-48h | **Frozen** | Hard | Emergency Only (High Cost) | Change Control Committee |
   800→| T < 48 hours | **Execution** | Fixed | None (break-fix only) | Operational Controller |
   801→
   802→**Key Insight:** The "Frozen Zone" prevents system nervousness by locking schedules close to execution.
   803→
   804→### Impact Scope by Change Type
   805→
   806→| Change Type | Example | Immediate Scope | Propagation Chain |
   807→|-------------|---------|-----------------|-------------------|
   808→| **Resource Capacity** | Man-hours drop 20% in March | Local (affected tasks) | Asset → Traffic → Service |
   809→| **Pit Unavailability** | Pit 3 closed for 2 weeks | Regional (depot) | Reschedule → Backlog → Ground trains |
   810→| **Urgent Task** | Train T045 breakdown | Preemption of lower priority | Cascade displacement |
   811→| **Constraint Change** | Annual budget cut 10% | Global (strategic) | Replanning required |
   812→| **Priority Change** | D-Overhaul becomes critical | Single asset | May displace others |
   813→| **Cascade Delay** | Task A delayed 2 days | Dependency chain | All successors shift |
   814→
   815→### Recalculation Strategies
   816→
   817→| Strategy | Trigger | Scope | Latency | Best For |
   818→|----------|---------|-------|---------|----------|
   819→| **Right-Shift** | Minor delay | Single resource | Milliseconds | Operational dispatch |
   820→| **Incremental Repair** | Resource failure | Regional (depot/route) | Seconds | Short-term rescheduling |
   821→| **Full Regeneration** | Major disruption | Global (network) | Minutes/Hours | Tactical planning / Crisis |
   822→| **Hybrid (Shadow Account)** | Fleet limits | Asset + Service | Seconds/Min | Complex rolling stock loops |
   823→
   824→### Constraint Propagation (CSP)
   825→
   826→When a constraint changes (e.g., task duration increases by 30 min):
   827→
   828→1. **Domain Reduction:** Earliest Start Time of dependent tasks increases
   829→2. **Arc Consistency:** System checks all connected dependencies
   830→3. **Chain Reaction:** Propagates until stable state (Fixed Point) or infeasibility (Empty Domain)
   831→4. **Lazy vs Eager:**
   832→   - **Eager:** Immediate propagation (always valid, but expensive)
   833→   - **Lazy:** Propagate only when queried or within time horizon (better performance)
   834→
   835→### Hierarchical Recalculation Order
   836→
   837→```
   838→Disruption occurs at Operational level (e.g., elevator broken)
   839→    │
   840→    ▼
   841→Level 1: Can we reschedule within current shift? (Incremental Repair)
   842→    │
   843→    ├── YES → Recalculate local sequence. Done.
   844→    │
   845→    └── NO → Escalate to Tactical
   846→            │
   847→            ▼
   848→        Level 2: Can we move to next week? (Slushy Zone)
   849→            │
   850→            ├── YES → Recalculate future workload.
   851→            │
   852→            └── NO → Escalate to Strategic/Emergency
   853→                    │
   854→                    ▼
   855→                Level 3: Use reserve train or cancel service?
   856→                    │
   857→                    └── Triggers Service Variation → Feeds back to strategic KPIs
   858→```
   859→
   860→### Edge Cases
   861→
   862→| Edge Case | Description | Mitigation |
   863→|-----------|-------------|------------|
   864→| **Infinite Loop** | Constraint A pushes task to 10:00, Constraint B pushes back to 9:00 | Visited state detection + Tabu Search |
   865→| **Infeasibility** | No solution satisfies all hard constraints | Relax soft constraints, show Minimal Conflict Set |
   866→| **Simultaneous Changes** | Two users change different things at same time | Event Sourcing with conflict resolution |
   867→| **Deadlock** | Circular dependency in pit allocation | Cycle detection in dependency graph |
   868→
   869→### Metrics for Recalculation Quality
   870→
   871→| Metric | Definition | Target |
   872→|--------|------------|--------|
   873→| **Nervousness** | How much the schedule changes after recalculation | Minimize |
   874→| **Absorption** | Ability to absorb small delays without rescheduling | Maximize |
   875→| **Recovery Time** | Time to return to original plan after disruption | Minimize |
   876→| **Passenger Impact** | Delays weighted by affected passengers | Minimize |
   877→
   878→### Implementation: Event Sourcing
   879→
   880→Instead of updating records in place, store each change as an immutable event:
   881→
   882→```typescript
   883→// Event types for change tracking
   884→type ScheduleEvent =
   885→  | { type: 'TaskScheduled'; taskId: string; startDate: Date; pit: string }
   886→  | { type: 'TaskRescheduled'; taskId: string; oldStart: Date; newStart: Date; reason: string }
   887→  | { type: 'ConflictDetected'; taskIds: string[]; conflictType: string }
   888→  | { type: 'ConstraintOverridden'; constraintId: string; overriddenBy: string }
   889→  | { type: 'ScenarioCreated'; scenarioId: string; baselineId: string }
   890→  | { type: 'ZoneStatusChanged'; from: ZoneStatus; to: ZoneStatus };
   891→```
   892→
   893→**Benefits:**
   894→- Full audit trail
   895→- "Time travel" to any past state
   896→- Simulate multiple futures (branching)
   897→- Essential for What-If scenarios
   898→
   899→### Data Type Implications for F104
   900→
   901→From this research, the data model must include:
   902→
   903→```typescript
   904→// Zone status for tasks
   905→type ZoneStatus = 'liquid' | 'slushy' | 'frozen' | 'execution';
   906→
   907→// Constraint classification
   908→type ConstraintType = 'hard' | 'soft';
   909→
   910→// On Task entity
   911→interface Task {
   912→  // ... existing fields ...
   913→
   914→  // Change propagation fields
   915→  frozenUntil?: Date;              // When task exits frozen zone
   916→  zoneStatus: ZoneStatus;          // Current zone
   917→  constraintType: ConstraintType;  // Hard or soft
   918→
   919→  // Dependency tracking
   920→  predecessors: string[];          // Tasks that must complete before this
   921→  successors: string[];            // Tasks that depend on this
   922→
   923→  // Rescheduling metadata
   924→  originalStartDate?: Date;        // For tracking drift
   925→  rescheduledCount: number;        // How many times moved
   926→  lastRescheduledAt?: Date;
   927→  lastRescheduledReason?: string;
   928→}
   929→
   930→// Scenario for What-If
   931→interface Scenario {
   932→  id: string;
   933→  name: string;
   934→  baselineId: string;
   935→  events: ScheduleEvent[];         // Event sourcing
   936→  metrics: {
   937→    nervousness: number;           // 0-1, lower is better
   938→    costDelta: number;             // vs baseline
   939→    conflictCount: number;
   940→    delayedTasks: number;
   941→  };
   942→}
   943→```

   955→
   956→### Continuous Human-in-the-Loop Optimization (CHiLO)
   957→
   958→The system should learn from planner interventions. If a planner consistently overrides a specific assignment suggested by the algorithm, the system should learn this "implicit constraint" and avoid such assignments in the future.
   959→
   960→From research on adaptive scheduling:
   961→> "Systems that incorporate feedback loops from user corrections can improve recommendation accuracy by 15-25% over 6 months of operation."
   962→
   963→### Generative AI in Planning
   964→
   965→Utilize Large Language Models (LLMs) to interpret natural language constraints:
   966→- "Don't schedule Class A trains on Track 4 this week due to nearby construction"
   967→- "Prioritize Train T045 for next available pit due to passenger complaints"
   968→
   969→LLMs can convert these into mathematical constraints for the solver, reducing the need for complex configuration interfaces.
   970→
   971→### Predictive Resilience
   972→
   973→Future systems will use Digital Twins and Event Sourcing to simulate the "Ripple Effect" *before* it occurs, transforming recalculation from an emergency reaction into an anticipated strategic decision.

   976→- Real-time mileage prediction with ML models
